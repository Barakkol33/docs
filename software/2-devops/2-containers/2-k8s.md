# Kubernetes (K8s) — Self-Learning Guide

This guide teaches you Kubernetes from the ground up. It assumes you already understand Docker (containers, images, networking). Each section introduces a concept, explains *why* it exists, and gives you hands-on commands to try.

---

## What is Kubernetes and why do you need it?

Docker lets you run containers on a single machine. But in production, you need more:
- What if the machine crashes? Your containers die with it.
- What if traffic spikes? You need more instances, fast.
- What if you have 50 services? You need automated deployment, networking, and monitoring.

Kubernetes solves these problems. It's a **container orchestration system** — you tell it *what you want* (e.g., "run 3 copies of my app"), and it figures out *how to make it happen* (scheduling, networking, restarting crashed containers, scaling).

The core idea is **declarative management**: you describe the desired state in YAML, and Kubernetes continuously works to match reality to that description. If a container crashes, Kubernetes restarts it. If a node dies, Kubernetes reschedules the workload elsewhere.

## Setting up a local cluster with `kind`

Before learning Kubernetes, you need a cluster. In production this would be managed by a cloud provider, but for learning you can run everything locally using **`kind`** (Kubernetes IN Docker) — it creates a Kubernetes cluster inside Docker containers on your machine.

### Create a cluster

```bash
kind create cluster --name demo
```

```text
Creating cluster "demo" ...
 • Ensuring node image (kindest/node:...)
 • Preparing nodes
 • Writing configuration
 • Starting control-plane
 • Installing CNI
 • Installing StorageClass
Set kubectl context to "kind-demo"
```

Kind created a single-node cluster and configured `kubectl` (the Kubernetes CLI) to talk to it.

### Verify it's working

```bash
kubectl cluster-info
kubectl get nodes -o wide
```

```text
NAME                 STATUS   ROLES           AGE   VERSION   INTERNAL-IP   OS-IMAGE
demo-control-plane   Ready    control-plane   45s   v1.29.2   172.18.0.2    Debian GNU/Linux 12 (bookworm)
```

You have one node with the `control-plane` role. In a real cluster you'd have multiple worker nodes too.

### Delete the cluster (when you're done)

```bash
kind delete cluster --name demo
```

## Architecture: how Kubernetes works under the hood

Understanding the architecture helps you debug problems later. Kubernetes has two main layers:

### Control plane (the "brain")

The control plane makes all the decisions. It runs these components:

- **kube-apiserver** — The front door to everything. Every `kubectl` command, every internal component, talks to the API server. It validates and processes requests.
- **etcd** — A distributed key-value store that holds all cluster state (what's deployed, where it's running, what the desired config is). Think of it as the cluster's database.
- **kube-scheduler** — When you create a pod, it doesn't have a node yet. The scheduler picks the best node based on resource availability, constraints, and affinity rules.
- **kube-controller-manager** — Runs a collection of controllers that watch cluster state and reconcile it. For example, the Deployment controller ensures the right number of pods exist. If a pod dies, the controller creates a replacement.

You can see these components running as pods:

```bash
kubectl get pods -n kube-system | grep -E "kube-apiserver|kube-scheduler|kube-controller-manager|etcd"
```

### Nodes (the "muscle")

Nodes are the machines (or containers, in kind's case) that actually run your workloads. Each node runs:

- **kubelet** — The node agent. It receives instructions from the control plane and ensures the assigned pods are running. If a container crashes, kubelet restarts it.
- **container runtime** — The software that actually runs containers (usually `containerd`). Kubelet tells it "start this container image" and it does.
- **kube-proxy** — Implements networking rules so that traffic to a Service reaches the right pods. It sets up iptables/ipvs rules on each node.
- **CNI plugin** — Handles pod-to-pod networking. In kind you'll see `kindnet`; in cloud environments you'll see cloud-specific plugins.

```bash
kubectl get pods -n kube-system | grep -E "kube-proxy|kindnet"
```

### Putting it together

When you run `kubectl create deployment web --image=nginx`, here's what happens:
1. `kubectl` sends the request to the **API server**.
2. The API server stores the desired state in **etcd**.
3. The **Deployment controller** sees a new Deployment and creates a ReplicaSet.
4. The ReplicaSet controller creates Pod objects.
5. The **scheduler** assigns each Pod to a node.
6. The **kubelet** on that node pulls the image and starts the container.

This chain — from desired state to running container — is the heart of Kubernetes.

## Kubernetes objects: the building blocks

Everything in Kubernetes is an **object** — a record of intent stored in etcd. You create objects (usually via YAML), and Kubernetes controllers work to make reality match.

### How objects are structured

Every Kubernetes object has three key sections:

- **`metadata`** — Identity: name, namespace, labels, annotations.
- **`spec`** — Desired state: what you want (e.g., "run 3 replicas of nginx").
- **`status`** — Observed state: what's actually happening (filled in by Kubernetes, not by you).

You can see this structure on any object:

```bash
kubectl get deploy -A -o yaml | head -n 30
```

The rest of this guide walks through the most important object types, building from simple to complex.

## Namespaces: organizing your cluster

Namespaces are like folders — they partition a cluster so different teams, projects, or environments don't collide. Resources in different namespaces can have the same name without conflict.

Every cluster starts with a few built-in namespaces:
- `default` — where things go if you don't specify a namespace.
- `kube-system` — system components (the ones we saw earlier).

Create your own namespace for this tutorial:

```bash
kubectl create namespace demo
kubectl config set-context --current --namespace=demo
kubectl get ns
```

```text
namespace/demo created
Context "kind-demo" modified.
NAME              STATUS   AGE
default           Active   10m
demo              Active   3s
kube-system       Active   10m
```

From now on, all commands will target the `demo` namespace by default. You can also create namespaces with YAML:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: demo
YAML
```

## Pods: the smallest unit

A **Pod** is the smallest thing Kubernetes deploys. It's one or more containers that share the same network (they can reach each other on `localhost`) and can share volumes.

In practice, most Pods contain a single container. Multi-container Pods are used for sidecars (e.g., a log shipper running alongside your app).

### Running a Pod directly

```bash
kubectl run nginx --image=nginx:stable --port=80
kubectl get pods -o wide
```

```text
pod/nginx created
NAME    READY   STATUS    RESTARTS   AGE   IP           NODE
nginx   1/1     Running   0          18s   10.244.0.6   demo-control-plane
```

Notice the pod got its own IP address (`10.244.0.6`). This is a **cluster-internal** IP — you can reach it from other pods, but not from your host machine (we'll fix that later with Services and port-forwarding).

Check its logs and inspect it:

```bash
kubectl logs pod/nginx --tail=5
kubectl exec -it pod/nginx -- sh -lc 'nginx -v && hostname && ip addr show eth0 | head'
```

**Important:** In production, you almost never create Pods directly. If a bare Pod crashes, nothing restarts it. Instead, you use a Deployment (covered next), which manages Pods for you.

Pod YAML equivalent:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:stable
      ports:
        - containerPort: 80
YAML
```

Note the `labels` field — this is a good time to understand labels and annotations, since they're everywhere in Kubernetes.

### Labels and annotations

**Labels** are key-value pairs attached to objects. They are the primary mechanism for **selection and grouping** — Services find Pods by label, Deployments manage Pods by label, and you filter `kubectl` output by label. Labels are meant to be queried and matched against.

```yaml
metadata:
  labels:
    app: nginx          # which application
    env: production     # which environment
    team: platform      # which team owns this
```

```bash
kubectl get pods -l app=nginx              # select by label
kubectl get pods -l app=nginx,env=prod     # multiple labels (AND)
kubectl get pods -l 'env in (dev,staging)' # set-based selector
```

**Annotations** are also key-value pairs, but they're for **metadata that isn't used for selection**. They store extra information for humans or tools — descriptions, config hints, tool-specific settings, links to dashboards, etc.

```yaml
metadata:
  annotations:
    description: "Main web frontend"
    prometheus.io/scrape: "true"
    kubectl.kubernetes.io/last-applied-configuration: "..."
```

**Rule of thumb:** If you need to select/filter by it → label. If it's just informational or for tooling → annotation.

## Deployments: running apps reliably

A **Deployment** is the standard way to run stateless applications. It provides:
- **Desired replica count** — "I want 3 copies of my app running."
- **Self-healing** — If a Pod dies, the Deployment creates a replacement.
- **Rolling updates** — When you change the image version, it gradually replaces old Pods with new ones (zero downtime).
- **Rollback** — If an update goes wrong, you can roll back to the previous version.

Under the hood, a Deployment creates a **ReplicaSet**, which in turn creates the Pods. The hierarchy is: Deployment → ReplicaSet → Pods.

### Create and scale a Deployment

```bash
kubectl create deployment web --image=nginx:stable
kubectl scale deployment/web --replicas=2
kubectl rollout status deployment/web
kubectl get deploy,rs,pods -l app=web
```

```text
deployment.apps/web created
deployment.apps/web scaled
deployment "web" successfully rolled out
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/web   2/2     2            2           35s
```

Now you have 2 nginx Pods managed by a Deployment. Try deleting one:

```bash
kubectl delete pod -l app=web --wait=false
kubectl get pods -l app=web
```

You'll see Kubernetes immediately creating a replacement — that's the self-healing in action.

Deployment YAML equivalent:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:stable
          ports:
            - containerPort: 80
YAML
```

A few things to understand in this YAML:
- `selector.matchLabels` tells the Deployment which Pods belong to it (by label).
- `template` is the Pod template — every Pod the Deployment creates will look like this.
- The labels in `template.metadata.labels` **must match** `selector.matchLabels`, otherwise the Deployment can't find its own Pods.

## Services: stable networking for Pods

Pods are ephemeral — they get created, destroyed, and rescheduled constantly. Each time, they get a new IP address. So how do other parts of your system find them?

A **Service** provides a stable endpoint (a fixed IP and DNS name) that routes traffic to the right Pods. It uses label selectors to find which Pods to send traffic to.

### Service types

- **`ClusterIP`** (default) — An internal virtual IP, reachable only inside the cluster. Most services use this.
- **`NodePort`** — Opens a port on every node in the cluster, making the service accessible from outside. Handy for local development.
- **`LoadBalancer`** — Provisions an external load balancer (cloud environments only).

### Create a Service

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
YAML
```

```bash
kubectl get svc web -o wide
```

```text
NAME   TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
web    ClusterIP   10.96.201.146   <none>        80/TCP    6s
```

The Service got a stable `CLUSTER-IP`. Any pod in the cluster can now reach nginx at `web.demo.svc.cluster.local:80` (or just `web` if it's in the same namespace).

### Accessing a Service from your machine

Since `ClusterIP` is internal-only, you can use `port-forward` to tunnel traffic from your host:

```bash
kubectl port-forward svc/web 8080:80
```

```text
Forwarding from 127.0.0.1:8080 -> 80
```

In another terminal:

```bash
curl -sS http://127.0.0.1:8080/ | head
```

### Headless Services

A **headless** Service sets `clusterIP: None`. Instead of a single virtual IP, DNS returns the individual Pod IPs. This is used with StatefulSets where you need to address specific pods (e.g., `db-0.db.demo.svc.cluster.local`).

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: web-headless
spec:
  clusterIP: None
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
YAML
```

### DNS in Kubernetes

Kubernetes runs CoreDNS, which gives every Service a DNS name:
- Full form: `<service>.<namespace>.svc.cluster.local`
- Short form (same namespace): `<service>`

This means your app code just connects to `db:5432` or `redis:6379` — no hardcoded IPs.

## Configuration: ConfigMaps and Secrets

Applications need configuration — database URLs, feature flags, API keys. Kubernetes provides two objects for this:

### ConfigMap (non-sensitive config)

A **ConfigMap** stores plain-text configuration. You can inject it into Pods as environment variables or mounted files.

```bash
kubectl create configmap app-config --from-literal=LOG_LEVEL=debug
kubectl get configmap app-config -o yaml
```

```text
configmap/app-config created
```

YAML equivalent:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "debug"
YAML
```

### Secret (sensitive config)

A **Secret** is like a ConfigMap but for sensitive data (passwords, tokens, keys). Values are base64-encoded in the YAML, and depending on cluster configuration, can be encrypted at rest in etcd.

```bash
kubectl create secret generic db-secret --from-literal=password='s3cr3t'
kubectl get secret db-secret
kubectl describe secret db-secret
```

```text
secret/db-secret created
NAME        TYPE     DATA   AGE
db-secret   Opaque   1      2s
```

YAML equivalent (using `stringData` so you don't need to base64-encode manually):

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  password: s3cr3t
YAML
```

**Note:** Kubernetes Secrets are not strongly encrypted by default — they're base64-encoded, which is not encryption. For real security, enable encryption at rest or use an external secret manager (e.g., Vault, SOPS, Sealed Secrets).

## Jobs and CronJobs: running tasks to completion

Not everything is a long-running service. Sometimes you need to run a one-off task (database migration, data export) or a scheduled task (nightly report, cleanup).

### Job (run once)

A **Job** creates a Pod that runs to completion and then stops. If the Pod fails, the Job can retry it.

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: batch/v1
kind: Job
metadata:
  name: hello
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: hello
          image: busybox:1.36
          command: ["sh","-lc","echo hello from job; sleep 1"]
YAML

kubectl wait --for=condition=complete job/hello --timeout=60s
kubectl logs job/hello
```

```text
job.batch/hello created
job.batch/hello condition met
hello from job
```

### CronJob (run on a schedule)

A **CronJob** is a Job that runs on a cron schedule. The syntax is the same as Unix cron.

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello-cron
spec:
  schedule: "*/5 * * * *"
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: Never
          containers:
            - name: hello
              image: busybox:1.36
              command: ["sh","-lc","date; echo hello from cronjob"]
YAML
```

This creates a new Job (and Pod) every 5 minutes. Each Job runs to completion independently.

## StatefulSets: stateful workloads

Deployments are great for stateless apps — every Pod is interchangeable. But databases and other stateful systems need:
- **Stable identities** — Pods get predictable names like `db-0`, `db-1`, `db-2` (not random hashes).
- **Stable storage** — Each Pod gets its own persistent volume that follows it across restarts.
- **Ordered startup/shutdown** — Pods are created and destroyed in order.

A **StatefulSet** provides all of this. It's always paired with a headless Service (so each Pod gets its own DNS entry).

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: db
spec:
  clusterIP: None
  selector:
    app: db
  ports:
    - port: 80
      targetPort: 80
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: db
spec:
  serviceName: db
  replicas: 1
  selector:
    matchLabels:
      app: db
  template:
    metadata:
      labels:
        app: db
    spec:
      containers:
        - name: nginx
          image: nginx:stable
          ports:
            - containerPort: 80
          volumeMounts:
            - name: www
              mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
    - metadata:
        name: www
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 1Gi
YAML
```

Key things in this YAML:
- `serviceName: db` links the StatefulSet to the headless Service.
- `volumeClaimTemplates` creates a separate 1Gi PersistentVolumeClaim for each replica. If you scale to 3 replicas, you get `www-db-0`, `www-db-1`, `www-db-2` — each with its own disk.

```bash
kubectl get statefulset,pods,pvc -l app=db
```

## DaemonSets: one Pod per node

A **DaemonSet** ensures that a copy of a Pod runs on every node (or a subset of nodes). This is used for infrastructure-level concerns:
- Log collectors (Fluentd, Fluent Bit)
- Monitoring agents (node exporter)
- Network plugins (kube-proxy, CNI)

You already have DaemonSets running in your cluster:

```bash
kubectl get daemonset -n kube-system
```

```text
NAME         DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
kindnet      1         1         1       1            1           <none>          10m
kube-proxy   1         1         1       1            1           <none>          10m
```

Here's an example DaemonSet. The `tolerations` allow it to run on the control-plane node too (which normally doesn't accept regular workloads):

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-sleeper
spec:
  selector:
    matchLabels:
      app: node-sleeper
  template:
    metadata:
      labels:
        app: node-sleeper
    spec:
      tolerations:
        - key: "node-role.kubernetes.io/control-plane"
          operator: "Exists"
          effect: "NoSchedule"
        - key: "node-role.kubernetes.io/master"
          operator: "Exists"
          effect: "NoSchedule"
      containers:
        - name: sleeper
          image: busybox:1.36
          command: ["sh","-lc","sleep 3600"]
YAML
```

When you add a new node to the cluster, the DaemonSet automatically schedules a Pod on it.

## Horizontal Pod Autoscaler (HPA): scaling by demand

An **HPA** automatically adjusts the number of Pod replicas based on metrics like CPU or memory usage. When load increases, it scales up; when load decreases, it scales down.

The HPA requires a **metrics pipeline** — usually `metrics-server` — to know the current resource usage.

```bash
kubectl autoscale deployment/web --cpu-percent=50 --min=1 --max=5
kubectl get hpa
```

```text
horizontalpodautoscaler.autoscaling/web autoscaled
NAME   REFERENCE        TARGETS         MINPODS   MAXPODS   REPLICAS   AGE
web    Deployment/web   <unknown>/50%   1         5         2          6s
```

If `TARGETS` shows `<unknown>`, it means metrics-server isn't installed or isn't ready yet. In kind, you may need to install it separately.

YAML equivalent:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 1
  maxReplicas: 5
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50
YAML
```

This says: "Keep average CPU utilization at 50%. Scale between 1 and 5 replicas as needed."

## Ingress: HTTP routing into the cluster

So far, we've used `port-forward` to access Services. In production, you use **Ingress** — an object that defines HTTP(S) routing rules (by hostname and path) to route external traffic to Services.

Ingress requires an **Ingress Controller** running in the cluster (e.g., nginx ingress controller, Traefik). The Ingress object is just config; the controller is what actually handles the traffic.

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web
spec:
  rules:
    - host: web.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web
                port:
                  number: 80
YAML
```

This says: "When a request comes in for `web.local`, route it to the `web` Service on port 80."

```bash
kubectl get ingress -A
```

Without an Ingress Controller installed, the Ingress object will exist but won't do anything. In kind, you can install the nginx ingress controller to test this.

## Receiving real traffic locally with MetalLB

So far we've used `port-forward` to access services — but that's a debugging tool, not how real traffic flows. In cloud environments, a `LoadBalancer` Service gets a real external IP from the cloud provider. Locally (kind, bare-metal), there's no cloud provider to hand out IPs. **MetalLB** fills that gap.

MetalLB is a load balancer implementation for bare-metal / local clusters. It assigns real IPs from a pool you define, so `LoadBalancer` Services actually work.

### Install MetalLB

```bash
kubectl apply -f https://raw.githubusercontent.com/metallb/metallb/v0.14.9/config/manifests/metallb-native.yaml
kubectl wait --namespace metallb-system --for=condition=ready pod --selector=app=metallb --timeout=90s
```

- YAML file is attached in `resources` directory.

### Configure an IP pool

First, find the Docker network range that kind uses:

```bash
docker network inspect kind -f '{{(index .IPAM.Config 0).Subnet}}'
# e.g., 172.18.0.0/16
```

Pick a small range within that subnet (e.g., `172.18.255.200-172.18.255.250`) and create the MetalLB config:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: demo-pool
  namespace: metallb-system
spec:
  addresses:
    - 172.18.255.200-172.18.255.250
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: demo-l2
  namespace: metallb-system
YAML
```

### Use it

Now create a `LoadBalancer` Service:

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: web-lb
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
  type: LoadBalancer
YAML
```

```bash
kubectl get svc web-lb
```

```text
NAME     TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)        AGE
web-lb   LoadBalancer   10.96.50.12    172.18.255.200   80:31234/TCP   5s
```

The `EXTERNAL-IP` is now a real routable IP (within your Docker network). You can hit it directly:

```bash
curl -sS http://172.18.255.200/
```

This is how traffic flows in production: client → LoadBalancer IP → Service → Pods. No port-forwarding needed.

### Making it feel real: local DNS with `/etc/hosts`

With a real IP from MetalLB and an Ingress, you can map a hostname to that IP in your local DNS so it feels like a real domain:

```bash
# Add to /etc/hosts (requires sudo)
echo "172.18.255.200 web.local" | sudo tee -a /etc/hosts
```

Now you can access your service by name:

```bash
curl -sS http://web.local/
```

This is exactly how it works in production — DNS resolves a hostname to a load balancer IP, which routes to your Ingress Controller, which routes to the right Service based on host/path rules.

To clean up when you're done:

```bash
# Remove the line from /etc/hosts
sudo sed -i '/web\.local/d' /etc/hosts
```

**Tip:** For managing many local domains, tools like `dnsmasq` can wildcard an entire domain (e.g., `*.local.dev → 172.18.255.200`) so you don't need to edit `/etc/hosts` for every service.

## `kubectl` cheat sheet

These are the commands you'll use every day. They're grouped by what you're trying to do.

### Viewing resources

```bash
kubectl get all                    # overview of common resources
kubectl get pods -o wide           # pods with IP and node info
kubectl get events --sort-by=.metadata.creationTimestamp | tail -n 20
```

### Debugging

```bash
kubectl describe deploy/web        # detailed info + events
kubectl describe svc/web
kubectl logs deploy/web --tail=20  # logs from a deployment's pods
kubectl logs pod/<pod> -c <container>  # specific container's logs
```

`describe` is your best friend for debugging. It shows events at the bottom — these tell you *why* something failed (image pull errors, insufficient resources, crash loops).

### Modifying resources

```bash
kubectl edit deploy/web            # opens in your editor
kubectl apply -f some.yaml        # apply a YAML file
kubectl diff -f some.yaml         # preview what would change
```

### Deleting resources

```bash
kubectl delete pod/nginx
kubectl delete ns demo             # deletes namespace and EVERYTHING in it
```

### Working with labels

Labels are how you organize, select, and filter resources:

```bash
kubectl get pods -l app=web        # filter by label
kubectl label pod nginx tier=frontend  # add a label
kubectl get pods --show-labels     # show all labels
```

### Getting a shell inside a Pod

```bash
kubectl exec -it deploy/web -- sh
kubectl exec -it pod/<pod> -- sh -lc 'env | sort | head'
```

### Accessing services locally

```bash
kubectl port-forward pod/nginx 8081:80
kubectl port-forward svc/web 8080:80
```

### JSON output and jq

Kubernetes outputs JSON natively. Combined with `jq`, you can extract exactly what you need:

```bash
# Get all pod names
kubectl get pods -o json | jq -r '.items[].metadata.name'

# Get pod name and status pairs
kubectl get pods -o json | jq -r '.items[] | "\(.metadata.name) \(.status.phase)"'

# Get all container images running in a namespace
kubectl get pods -o json | jq -r '.items[].spec.containers[].image' | sort -u

# Get events for a specific pod
kubectl get events -o json | jq '.items[] | select(.involvedObject.name=="web-abc123") | .message'

# Extract a secret value (base64 decode)
kubectl get secret db-secret -o json | jq -r '.data.password' | base64 -d

# Get resource requests/limits for all containers
kubectl get pods -o json | jq '.items[] | {pod: .metadata.name, containers: [.spec.containers[] | {name, resources}]}'
```

`jq` is invaluable for scripting and automation — any time you need to extract data from Kubernetes for a script or report, `-o json | jq` is the way.

## Helm: package management for Kubernetes

Writing YAML for every resource gets tedious, especially for complex applications. **Helm** is a package manager for Kubernetes.

The idea is simple: someone already wrote all the YAML templates needed to deploy an application (Deployment, Service, ConfigMap, Ingress, etc.) and packaged them into a **Chart**. You install that chart into your cluster, and all the resources get created for you. You customize the deployment by editing a single `values.yaml` file (or passing `--set` flags) — no need to touch the templates themselves.

- A **Chart** is the package — a collection of templated YAML manifests.
- A **Release** is an installed instance of a chart in your cluster.
- A **values.yaml** is how you configure a chart — replica count, image version, resource limits, feature toggles, etc.

### Common Helm workflow

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm install my-nginx bitnami/nginx --namespace demo --create-namespace
helm list -n demo
helm upgrade my-nginx bitnami/nginx -n demo
helm uninstall my-nginx -n demo
```

```text
NAME       NAMESPACE  REVISION  UPDATED                                STATUS    CHART      APP VERSION
my-nginx   demo       1         2026-02-14 10:40:12 +0000 UTC         deployed  nginx-...   1.27.0
```

Helm also supports `values.yaml` files, letting you customize chart behavior without editing templates directly.

### Creating your own Helm chart

When you want to package your own application as a chart:

```bash
helm create myapp
```

This scaffolds a chart directory:

```text
myapp/
  Chart.yaml          # chart metadata (name, version, description)
  values.yaml         # default configuration values
  templates/          # Kubernetes manifest templates
    deployment.yaml
    service.yaml
    ingress.yaml
    hpa.yaml
    serviceaccount.yaml
    _helpers.tpl      # reusable template snippets
    NOTES.txt         # post-install message shown to the user
  charts/             # dependency charts (sub-charts)
```

**Chart.yaml** defines the chart itself:

```yaml
apiVersion: v2
name: myapp
description: My application Helm chart
version: 0.1.0        # chart version (bump on chart changes)
appVersion: "1.0.0"   # version of the app being deployed
```

**values.yaml** is where you put all configurable knobs. Users override these without touching templates:

```yaml
replicaCount: 2
image:
  repository: myuser/myapp
  tag: "1.0.0"
  pullPolicy: IfNotPresent
service:
  type: ClusterIP
  port: 80
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    memory: 256Mi
```

**Templates** use Go templating to reference values. For example, `templates/deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      app: {{ include "myapp.name" . }}
  template:
    metadata:
      labels:
        app: {{ include "myapp.name" . }}
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          ports:
            - containerPort: {{ .Values.service.port }}
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
```

Key templating patterns:
- `{{ .Values.x }}` — reads from `values.yaml`.
- `{{ .Chart.Name }}` — reads from `Chart.yaml`.
- `{{ include "myapp.fullname" . }}` — calls a helper from `_helpers.tpl`.
- `{{- toYaml ... | nindent N }}` — renders a YAML block with proper indentation.

### Developing and testing a chart

```bash
# Render templates locally (see what YAML would be generated, without applying)
helm template myapp ./myapp

# Render with custom values
helm template myapp ./myapp -f custom-values.yaml

# Lint the chart for errors
helm lint ./myapp

# Install from local directory
helm install myapp ./myapp --namespace demo

# Upgrade after making changes
helm upgrade myapp ./myapp --namespace demo

# Override values on the command line
helm install myapp ./myapp --set replicaCount=3 --set image.tag=v2.0.0

# Dry-run to see what would be applied
helm install myapp ./myapp --dry-run --debug
```

### Packaging and sharing

```bash
# Package into a .tgz file
helm package ./myapp
# myapp-0.1.0.tgz

# Push to an OCI registry (Helm 3.8+)
helm push myapp-0.1.0.tgz oci://registry.example.com/charts
```

The typical flow in CI/CD: `helm lint` → `helm template` (validate) → `helm package` → `helm push` to a registry → `helm upgrade --install` in the deployment stage.

## Persistent storage: StorageClass and PersistentVolumeClaims

Containers are ephemeral, so Kubernetes provides a storage abstraction with three layers:

### StorageClass (how storage is provisioned)

A **StorageClass** defines *what kind* of storage is available and *how* it gets created. It's like a template: "when someone asks for storage, use this provisioner to create it with these settings."

```bash
kubectl get storageclass
```

```text
NAME                 PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE
standard (default)   rancher.io/local-path   Delete          WaitForFirstConsumer
```

In kind, the default StorageClass uses local-path provisioner (storage on the node's disk). In cloud environments, StorageClasses map to cloud disk types — e.g., `gp3` EBS volumes on AWS, `pd-ssd` on GCP. You don't usually create StorageClasses yourself; they come with the cluster or are set up by the platform team.

When a PVC doesn't specify a StorageClass, Kubernetes uses the one marked `(default)`.

### PV and PVC (requesting and using storage)

- **PersistentVolume (PV)** — A piece of storage provisioned in the cluster (like a physical disk). With dynamic provisioning (via StorageClass), PVs are created automatically — you rarely create them manually.
- **PersistentVolumeClaim (PVC)** — A request for storage from a Pod. Kubernetes binds it to an available PV (or dynamically creates one via the StorageClass).

```bash
cat <<'YAML' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: demo-pvc
spec:
  accessModes: ["ReadWriteOnce"]
  resources:
    requests:
      storage: 1Gi
YAML
```

Because kind has a default StorageClass, this PVC will be automatically provisioned. In cloud environments, PVCs typically map to cloud disks (EBS, Persistent Disks, etc.).

## Concepts you'll encounter in real clusters

As you move beyond local development, you'll encounter these:

- **Operator pattern** — A controller that encodes app-specific operational knowledge. For example, a PostgreSQL operator knows how to handle backups, failovers, and scaling. Operators usually define Custom Resource Definitions (CRDs) — new object types specific to their domain.
- **Sidecar container** — An extra container in the same Pod that adds capabilities: a proxy, a log shipper, a cert refresher. The sidecar shares the Pod's network and volumes.
- **Istio** — A service mesh that adds traffic management, mutual TLS, and observability via sidecar proxies injected into every Pod.
- **cert-manager** — Automates TLS certificate issuance and renewal (e.g., from Let's Encrypt).
- **Karpenter** — Node autoscaler that provisions new nodes when pods can't be scheduled (primarily cloud environments).

### k9s: terminal UI for Kubernetes

**k9s** is an interactive terminal-based UI for Kubernetes. Instead of typing `kubectl` commands repeatedly, you get a live, navigable dashboard of your cluster. Highly recommended for day-to-day work.

**Quick start:**

```bash
k9s                        # launch (uses current kubectl context)
k9s -n demo                # launch in a specific namespace
k9s --context kind-demo    # launch with a specific context
```

**Essential navigation:**

| Key | Action |
|---|---|
| `:pods` | Switch to Pods view (type any resource: `:deploy`, `:svc`, `:ns`, `:jobs`, etc.) |
| `/` | Filter/search within the current view |
| `Enter` | Inspect selected resource (YAML) |
| `l` | View logs for selected pod |
| `s` | Shell into selected pod |
| `d` | Describe selected resource |
| `ctrl-d` | Delete selected resource |
| `Esc` | Go back |
| `:q` | Quit |

k9s auto-refreshes, so you can watch pods start, crash, and restart in real time — much faster than re-running `kubectl get pods`.

**Resources:**
- Documentation: https://k9scli.io/
- Key bindings reference: https://k9scli.io/topics/commands/
